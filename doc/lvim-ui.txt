1. INTRODUCTION                                            *lvim-ui-introduction*

lvim-ui is the floating UI toolkit of the lvim-tech set: the surface / frame /
button / bar chassis plus a thin presenter (select / multiselect / input /
confirm / tabs / info). Every lvim-tech plugin that draws a window builds on it,
so the whole set frames identically. It is a pure toolkit — it renders windows
and owns no feature of its own.

==============================================================================
2. REQUIREMENTS                                            *lvim-ui-requirements*

- Neovim >= 0.12.x
- lvim-utils (palette / highlight / cursor / merge helpers)
- lvim-hud (optional) — a surface title can publish to the statusline overlay.

==============================================================================
3. INSTALLATION                                            *lvim-ui-installation*

lvim-installer (recommended) — open the Plugins tab and install it: >vim
    :LvimInstaller plugins
<
Native (vim.pack): >lua
    vim.pack.add({
        { src = "https://github.com/lvim-tech/lvim-utils" },
        { src = "https://github.com/lvim-tech/lvim-ui" },
    })
    require("lvim-ui").setup({})
<

4. USAGE                                                          *lvim-ui-usage*

>lua
    local ui = require("lvim-ui")
    ui.select({ title = "Pick", items = { "one", "two" } }, function(ok, index) end)
    ui.multiselect({ title = "Toggle", items = { "a", "b" } }, function(ok, sel) end)
    ui.input({ title = "Name" }, function(ok, value) end)
    ui.confirm({ title = "Delete?" }, function(yes) end)
    ui.tabs({ title = "Settings", tabs = {} }, function(ok, result) end)
    local win = ui.info({ "read-only" }, { title = "Info" })
<
`ui.tabs` can additionally host a PREVIEW panel beside the tab content: pass
`preview = <provider>` (a surface content provider, typically built on
`require("lvim-ui.preview").new({ item = … })`) and an optional
`preview_side = "right"|"left"|"above"|"below"`. The block plugs into the
chassis preview machinery — <Tab> / <C-l> move between the panels, <C-e>
hides the preview, <C-n>/<C-p> rotate its side, and <C-d>/<C-u> scroll it half
a screen WITHOUT leaving the list (the fzf-lua / Magit model — a preview panel
hides its cursor, so this is the only way to read past its first screen without
spending a <Tab>).

The low-level chassis is `require("lvim-ui.surface")` (framed floating / docked
windows) with `require("lvim-ui.button")` / `require("lvim-ui.bar")` for
navigable button bars.

                                                                    *lvim-ui-mouse*
Mouse ~

Every keyboard-activated element is also a left-click target — a click does
exactly what pressing its key does, and the keyboard behaviour is unchanged
(click is additive; a no-op while 'mouse' is empty). It is built into the
primitives, so every consumer gets it for free:

- Header / footer / tab / filter bars — click a button to fire it, a tab header
  to switch to it, a filter chip to apply it. Hit-testing uses the bar's own
  rendered spans, so a click that hits no button is ignored.
- select / multiselect lists — click a row to focus it and confirm (select) or
  toggle its checkbox (multiselect), as <CR> / <Space> do on the focused row.
- tabs / form rows — click a settings row to activate it per its type (toggle a
  boolean, cycle a select, expand an accordion, run an action / menu item, open
  a text/number editor); click a toolbar-row button to run it.

- |lvim-ui.tree()| rows — click a row to select + activate it (a collapsed
  foldable node expands first); click the fold chevron — or double-click the
  row — to toggle the fold.

The |lvim-ui.menu()| primitive has NO click support: it is a non-focusable,
passive insert-mode overlay with no window to route a click to, so accepting a
completion stays a keyboard action by design.

                                                                 *lvim-ui.menu()*
require("lvim-ui").menu({opts})                        → handle

The cursor-anchored NON-FOCUSABLE popup primitive (completion menus): a passive
projection redrawn per keystroke while focus stays in the editing buffer. One
long-lived window/buffer per handle (`focusable = false`, `noautocmd`,
repositioned via |nvim_win_set_config()| — never recreated); the window is glued
to `anchor = { lnum, col }` (the matched keyword's START, so it does not shift
while typing) and flips below/above per `direction_priority = { "s", "n" }`
near the screen edge. Rows are BOX lists (lead box + label + `right = true`
detail); ALL row colour — box highlights, LAZY matched-char `positions`
callbacks, the scrollbar — is applied by a decoration provider as EPHEMERAL
extmarks on VISIBLE lines only. A row may carry a `hl` (a full-row background,
painted under everything — tint each row by category) and a `sel_hl` (the
full-row background while THIS row is selected, used instead of the shared
selection group — a per-row selection colour). Selection is a full-row
`hl_eol` background (`LvimUiMenuSel` by default, or the row's `sel_hl`) painted
by the decoration provider — as reliable as the box spans, and NOT `cursorline`
(a current-window feature this passive window can't use). A box with its OWN bg
may pass its own `sel_hl`, the group used for it while its row is SELECTED, so
it re-tints against the selection bar instead of punching a hole in it. A
sibling docs slot
(`docs_show(lines, { filetype })` / `docs_hide()` / `docs_scroll(delta)` /
`docs_visible()`) docks a second non-focusable window FLUSH beside the menu
(east, flipping/shrinking west near the edge) behind the canonical
inter-panel divider (`config.separator` — the same `│` rule the surface
chassis draws between side-by-side panels), so no buffer text bleeds between
the two panels.

Options: `max_height` (12), `max_width` (60), `min_width` (0), `col_offset`
(0), `direction_priority` ({ "s", "n" }), `scrollbar` (true), `zindex` (65),
`docs` ({ max_width = 80, max_height = 20 }), `hl` (group overrides).

Handle: `show({ items, anchor, selected })`, `update(items, selected)`,
`move(anchor)`, `select(i)`, `select_move(delta)`, `selected()`, `count()`,
`visible()`, `hide()`, `close()`, `docs_show()`, `docs_hide()`,
`docs_scroll(delta)`, `docs_visible()`, `win()`.

Groups: `LvimUiMenuNormal`, `LvimUiMenuSel`, `LvimUiMenuMatch`,
`LvimUiMenuDetail`, `LvimUiMenuThumb`, `LvimUiMenuTrack` — palette-bound; the
selection bar and the scrollbar tint over the PANEL shade.

                                                                 *lvim-ui.hint()*
require("lvim-ui").hint({opts})                        → handle

The NON-FOCUSABLE key-hint BAR: one full-width row pinned above the statusline
that announces the live keys while a modal SUB-MODE owns the keyboard (an
interactive resize / move loop). The sanctioned surface for a key hint — never
an `echo`, never a hand-rolled float.

Focus never leaves the user's real window (the sub-mode's keys act ON it, and it
repaints under an explicit `redraw` inside a blocking `getcharstr()`), so — like
|lvim-ui.menu()| — it is a passive projection, not a chassis modal. Its items are
ordinary bar records (`{ key, name, style?, hl? }`, or a `separator`) rendered
through the shared `ui.surface.button` + `ui.bar` box model, so a hint key badge
is byte-identical to a footer one and the overflow chevrons come for free.

Options: `align`, `default_style`, `fill_hl`, `zindex`, `filetype` — each
defaulting to `config.hint`.

Handle: `show(items)`, `update(items)` (same window, no flicker), `hide()`,
`close()`, `visible()`.

                                                                 *lvim-ui.tree()*
require("lvim-ui").tree({opts})                        → handle

The generic node-provider TREE — the shared content layer every lvim-tech tree
panel renders through (the lvim-files file tree, the lvim-lsp outline), instead
of each hand-rolling fold state + indent guides + markers + extmarks. The tree
is ONLY the content: the handle's `provider` is a surface content provider the
consumer plugs into its OWN `surface.open` (a persistent native split, a modal
float, a provider tab) — the chassis keeps owning the window, dock, cursor
hiding, footer and teardown.

Node contract (lazy or eager): >lua
    {
        id = "stable-id",       -- fold state / focus / mark are keyed by it
        label = "name",
        icon = "",             -- lead kind icon (Nerd Font)
        icon_hl = "Directory",  -- or `hl` = ONE group for icon AND label
        label_hl = "Normal",    --   (source-colour mode; wins over both)
        kind = "dir",           -- free-form consumer tag
        expandable = true,      -- chevron before children are known (lazy)
        children = function(node) return nodes end, -- or an eager list;
                                --   a function runs only while EXPANDED,
                                --   once per render (live re-decoration)
        detail = "signature",   -- dim eol virtual text
        badges = { { "M ", "DiffAdd" } }, -- right-aligned virt-text cells
        actions = { d = function(node, t) end }, -- per-node extra keys
        data = anything,        -- consumer payload (selected()/callbacks)
    }
<
Options: `root` (nodes | one node | a FACTORY re-run per render),
`default_expanded` (false; true = an outline), `connectors` (false; ├/└ on leaf
rows), `elide_guides` (true; false = a solid │ per level, the file-tree style),
`margin` (0), `icons` ({ fold_open, fold_closed, guide, branch, branch_last }),
`hl` ({ guide, fold, detail, mark, empty, thumb, track } group overrides),
`empty` (" No entries"), `header` (fun(width) → lines, spans — static rows
above the tree, e.g. a root band; the cursor is kept off them), `filetype`
(stamped on the buffer — the cursor `panel_ft` registration), `cursorline`
(true), `hide_cursor` (false; modal usage), `scrollbar` (true — a right-edge
thumb when the tree overflows), `size` (fun() → w, h override), `keys`
({ activate = { "l", "<CR>" }, collapse = "h" }; false = none), and the hooks
`on_activate(node, t)`, `on_expand(node, t)`, `on_collapse(node, t)`,
`on_move(node|nil, t)`, `on_keys(map, pan, st, t)` (the consumer's own keymaps,
bound AFTER the canonical ones so a config key overrides on a clash),
`on_render(t)`, `on_close(pan)`.

Handle: `provider` (plug into `surface.open{ content.blocks }`),
`set_root(nodes|node|fun)`, `render()` (sync), `refresh()` (coalesced),
`selected()`, `node_at(line)`, `row_of(id)`, `visible()`, `focus(id)`,
`expanded(id)`, `expand(id)`, `collapse(id)`, `toggle(id)`, `expand_all()`,
`collapse_all()`, `all_expanded()`, `set_expanded(map)` (bulk fold replace —
the accordion/auto-fold seam; no hooks, no render), `expanded_state()`,
`get(id)`, `mark(id, { move_cursor })` (the follow-row tint an outline uses to
track the source cursor), `expand_or_activate()`, `collapse_or_parent()` (the
canonical `l`/`h`, exposed for consumer keymaps), `buf()`, `win()`, `valid()`.

Built in, identically for every consumer: the canonical keys, per-node
`actions` (bound lazily as they appear; a consumer's own key wins), the mouse
canon (row click = select + activate; chevron click / double-click = fold
toggle; hide-cursor modals route through the chassis `on_click` seam), and a
right-edge scrollbar painted as EPHEMERAL decoration-provider extmarks on
visible rows only (the menu canon).

Groups: `LvimUiTreeGuide`, `LvimUiTreeFold`, `LvimUiTreeDetail`,
`LvimUiTreeMark`, `LvimUiTreeEmpty`, `LvimUiTreeThumb`, `LvimUiTreeTrack` —
palette-bound; per-tree overrides via `hl`.

5. CONFIGURATION                                          *lvim-ui-configuration*

                                                               *lvim-ui.setup()*
require("lvim-ui").setup({opts})

Merges {opts} into the live config in place; every reader
(`require("lvim-ui.config")`) sees the effective values. Optional — the defaults
work as-is. See the README for the FULL default config; the option set is
documented as `LvimUiConfig` in `lua/lvim-ui/config.lua` (border, content_border,
separator, group_border, chevrons, size, backdrop, title_line, counter,
title_pos, tint, icons, labels, keys, …).

A centred float popup is MODAL by default (`trap_focus = true`): while it is open,
focus cannot leave it — a `<C-w>` window jump or a mouse click on another window
bounces straight back, so the only way out is the popup's own keys (`q`/`<Esc>`/an
action). Docked / hosted panels (which set `host`/`position`/`on_escape_*`) coexist
with the editor and never trap. Set `trap_focus = false` to disable globally, or on
a single `surface.open` to opt that popup out.
